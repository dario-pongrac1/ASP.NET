## Plan: Lab2 UI za sustav naručivanja kod mehaničara
Plan je usklađen s tvojim odlukama:
- svi 8 entiteta
- paper work-order vizualni smjer
- custom UX sub-agent datoteke u repozitoriju

**Koraci**

1. Faza 1: Stabilizacija arhitekture za Lab2  
Postaviti read-only MVC tok Controller -> mock repository/service -> ViewModel -> Razor view, uz zadržavanje postojećih seed podataka. Ovo je preduvjet za sve ostalo.

2. Faza 1: Mock repository sloj nad postojećim contextom (ovisno o 1)  
Za svaki entitet definirati read-only dohvat (GetAll, GetById i relacije) tako da controlleri ne rade direktno s listama.

3. Faza 1: Startup i DI čišćenje (ovisno o 2)  
Ažurirati registracije u Program.cs i prilagoditi Lab1 console ispis da ne smeta web demonstraciji.

4. Faza 2: Index + Details za svih 8 entiteta (paralelno u 2 trake)  
Traka A: Customer, Vehicle, Mechanic, ServiceOrder.  
Traka B: ServiceCategory, ServiceItem, OrderLine, AppointmentSlot.  
Svaki Details mora imati linkove prema povezanim entitetima.

5. Faza 2: ViewModel sloj (paralelno s 4 nakon prvih controllera)  
Napraviti view-modele za list/detail prikaze i izvući prikazne vrijednosti (status label, owner full name, sažeci).

6. Faza 3: Kompletna navigacija + breadcrumbs (ovisno o 4)  
Proširiti layout navigaciju za svih 8 entiteta, dodati list->details i details->details linkove, te zajednički breadcrumbs partial za sve stranice.

7. Faza 3: Unique UX, non-standard (ovisno o 6)  
Primijeniti paper work-order stil: svijetla “radni nalog” podloga, tehnički grid/linije, pečat-badge elementi, tipografija koja nije default bootstrap, i responzivnost mobile+desktop.

8. Faza 3: Specifična custom stranica (ovisno o 4)  
Izraditi domenski home/dashboard (npr. Daily Workshop Board: današnji termini, hitne narudžbe, opterećenje mehaničara, brzi linkovi).

9. Faza 4: Custom UX sub-agent u repo (može krenuti nakon 1)  
Dodati agent definition i prompt/instructions u .github strukturu repoa s jasnim UX pravilima (stil, komponente, layout principi).

10. Faza 4: Dokaz da glavni agent poziva UX sub-agenta (ovisno o 9)  
Kod UI izmjena koristiti workflow koji eksplicitno spawna UX sub-agenta i sačuvati log/screenshot kao dokaz za bodovanje.

11. Faza 5: Verifikacija i priprema za usmeno (ovisno o 1-10)  
Proći rubricu, ručno testirati sve rute/navigaciju/breadcrumbs, provjeriti jedinstvenost UX-a, i pripremiti kratko objašnjenje arhitekture + promptanja agenata.

- da dobije samo podatke koji su mu stvarno potrebni

- da ne “prelijevaš” cijeli domain model u UI

- Entity = podatak iz domene / spremnika  
- ViewModel = podatak pripremljen za prikaz u view-u

**DI (Dependency Injection)**
DI znači da objekt ne kreira sam svoje ovisnosti (`new ...`), nego ih dobije izvana kroz container.

U tvom projektu:
1. Registracija je u Program.cs:
- Program.cs: `AddSingleton<WorkshopDataContext>()`
- Program.cs: `AddScoped<IWorkshopReadRepository, WorkshopReadRepository>()`
2. Korištenje je u kontrolerima kroz konstruktor, npr. HomeController.cs prima `IWorkshopReadRepository`.

Zašto se koristi:
1. Manja sprega između slojeva (controller ovisi o ugovoru, ne konkretnoj klasi).
2. Lakša zamjena implementacije (npr. sutra drugi repository).
3. Lakše testiranje (mock/fake implementacije).
4. Centralna kontrola životnog vijeka objekata (`Singleton`, `Scoped`, `Transient`).

U tvom slučaju:
- `WorkshopDataContext` je singleton jer drži in-memory seed podatke koje dijeli cijela aplikacija.
- Repository je scoped jer je vezan uz jedan request.

**Što je Razor**
Razor je view engine za `.cshtml` datoteke: miješa HTML i C# (`@model`, `@foreach`, `@if`) i na runtime/compile-time to prevodi u C# kod koji renderira HTML response.

Primjer: Index.cshtml koristi `@model WorkshopDashboardViewModel` i petlju za termine.

**Zašto postoji Shared/Layout i čemu služi**
_Layout.cshtml je zajednički “okvir” svih stranica:
1. `<head>` (CSS/JS linkovi)
2. top navigacija
3. zajednički wrapper (`app-shell`)
4. skripte na dnu

Unutar layouta se poziva `@RenderBody()`, i tu ulazi sadržaj konkretne stranice (npr. Home, Vehicles, Mechanics).

Render slijed (po requestu):
1. Ruta pogodi akciju kontrolera.
2. Akcija vrati `View(...)`.
3. Razor pronađe view.
4. Primijeni `_ViewStart` (odredi layout).
5. Renderira view sadržaj.
6. Ubaci ga u layout na `@RenderBody()`.
7. Renderira opcionalne sekcije (`@RenderSectionAsync`).

**_ViewImports i _ViewStart**
1. _ViewImports.cshtml:
- dodaje common `@using`
- uključuje Tag Helpere (`@addTagHelper`)
- vrijedi za sve viewove ispod te mape
- koristi se pri kompilaciji/renderiranju viewa

2. _ViewStart.cshtml:
- izvršava se prije svakog view rendera
- u tvom projektu postavlja `Layout = "_Layout";`
- zato sve stranice automatski koriste zajednički layout bez ručnog postavljanja u svakom viewu

**Čemu služi site.css i gdje se koristi**
site.css je glavni globalni stylesheet:
1. definira theme varijable (`:root`)
2. stilizira nav, hero, tablice, breadcrumbs, responsive ponašanje
3. daje cijeli “dark automotive” izgled

Uključuje se u _Layout.cshtml, pa se automatski primjenjuje na sve stranice koje koriste layout.

User: cemu ovo sluzi?

GitHub Copilot: Ovaj dio u _Layout.cshtml služi kao “okvir” svake stranice.

Što radi svaka linija:

1. `<div class="container app-shell"> ...`
- zajednički wrapper za sadržaj
- daje širinu, padding i vizualni okvir (stilovi su u site.css)

2. `<main role="main" class="pb-3">`
- semantički glavni sadržaj stranice
- `role="main"` pomaže accessibility alatima
- `pb-3` daje donji razmak (Bootstrap utility class)

3. `@RenderBody()`
- najvažnije: tu se ubacuje sadržaj konkretne view stranice
- npr. kad otvoriš Home, ovdje se umetne Index.cshtml

4. `<script src="~/lib/jquery/dist/jquery.min.js"></script>`
- učitava jQuery biblioteku (ako je koristiš u JS-u)

5. `<script src="~/lib/bootstrap/dist/js/bootstrap.bundle.min.js"></script>`
- učitava Bootstrap JS (dropdown, collapse navbar, modal itd.)

6. `<script src="~/js/site.js" asp-append-version="true"></script>`
- učitava tvoj custom JS
- `asp-append-version="true"` doda verzijski query string zbog cache bustinga

7. `@await RenderSectionAsync("Scripts", required: false)`
- omogućuje da pojedinačni view po potrebi doda svoje dodatne skripte
- `required: false` znači da view ne mora imati tu sekciju

Praktično: layout daje zajednički kostur, `RenderBody()` ubacuje sadržaj stranice, a skripte na dnu učitavaju funkcionalnost.
